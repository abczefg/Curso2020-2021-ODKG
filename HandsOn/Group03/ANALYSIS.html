1. ANALYSIS OF THE DATA SOURCE

The data source is a CSV file containing information about public contracts awarded by Public Works and Government Services Canada (PWGSC) during the 2019-2020 fiscal year.

The dataset provides information about 23286 public contracts between Government entities and private companies from Canada and the United States. For each contract, data about the contract, the supplier company and the contracting entity is provided. In total, 46 variables are available for each contract.

Before developing our ontology, we observe that some of the variables consist on french translations of others, which we are not going to consider.  Also some variables, such as "amendment-number" or "number-records", will be removed since they are not relevant to the purpose of our application. When cleaning the dataset we will also pay attention to those contracts for which not enough information is provided (with empty or N/A cells), in order to remove them from our application.

We will focus on the main properties of the contract such as its value and the field it belongs to. We will link this data to both the supplier and the contracting parts, and also extract the basic information of these entities: their name, area/department, and location data (from their postal code to their country).

  1. DEFINE RESOURCE NAMING STRATEGY.

The names of the columns were not appropriate for an ontology, so we renamed almost every class so the graph has a better shape for an ontology definition.

We will use the ontology from “schema.org” because it provides terms and properties for public places that we can use for our City, PostalAddress, Province/State, and Country entities. Place will be in charge of managing those entities.

We named the final user of the contracts as EntitiyEndUser, we discussed that this entity is specific in this context and it won’t have an ontology defined for it, so we have to make it for this entity.

We discussed if the entity Supplier would use the ontology VCard for naming itself, however we decided that it should not use it for just one property. Supplier has other propertie which is hasEmployee, we will create this property in our ontology because it just manages the number of employees that the supplier has and we didn’t find an ontology that has this information as property at first.

Finally for the Contract entity we investigated and decided to create all the properties in our own domain. We find this easier to implement and to follow for this learning purpose.
